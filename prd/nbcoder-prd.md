## 1. 产品概述

### 1.1 产品定位

**NBCoder** 是一款私有化部署的 AI 驱动软件开发平台。用户通过自然语言输入需求，平台自动完成需求结构化、技术设计、任务拆解、代码开发、测试验证的完整开发流水线。

**核心理念**: 用 Vibe 的方式输入，用工程的方式产出。

### 1.2 核心价值

- **Vibe 输入，结构产出**: 需求可以随口说、碎片化输入，AI 实时整理成结构化需求卡片
- **知识驱动**: 项目知识库和开发规范贯穿全流程，让 AI 理解你的项目上下文
- **卡片式需求管理**: 每个需求是一张独立卡片，有状态、有依赖、可追溯
- **7 阶段流水线**: 需求分析 → 方案设计 → 任务拆解 → 测试用例 → 代码开发 → 测试验证 → 评审合并
- **Agent 自动执行**: 需求拆解为可执行任务后，Agent 矩阵自动完成代码开发
- **文档血缘追溯**: 所有产出物可追溯上下游关系

### 1.3 目标用户

- 个人开发者：需要结构化开发流程的独立开发者
- 小型团队：3-5 人开发团队，需要 AI 辅助开发
- 技术创业者：快速验证想法的早期项目

### 1.4 部署形态

私有化部署，单一租户，无需多租户隔离。

### 2.5 AI Provider 配置

NBCoder 支持配置多个 AI Provider，每个 Provider 可包含多个模型。通过 Agent 模型链配置，系统自动为不同类型的 Agent 分配合适的模型。

#### 2.5.1 核心能力

- **多 Provider 管理**：支持 OpenAI、Claude、本地模型等多种 Provider
- **多模型支持**：每个 Provider 可配置多个模型变体
- **智能路由**：根据 Agent 类型自动选择合适的模型
- **成本优化**：不同任务使用不同性价比的模型

#### 2.5.2 Provider 配置

每个 Provider 需要配置：

| 配置项 | 说明 |
|--------|------|
| Provider 名称 | 显示名称 |
| Provider 类型 | OpenAI / Anthropic (Claude) / OpenAI 兼容（本地模型） |
| API Endpoint | Provider 地址，本地模型必填 |
| API Key | 认证密钥（加密存储） |

每个模型需要配置：

| 配置项 | 说明 |
|--------|------|
| 模型 ID | Provider 提供的模型标识符 |
| 上下文窗口 | 最大 Token 数 |
| 是否默认 | 每 Provider 只能有一个默认模型 |

#### 2.5.3 Agent 模型链

模型链定义每类 Agent 执行时使用的模型：

| Agent 类型 | 用途 | 推荐模型 |
|------------|------|---------|
| 产品 Agent | 用户对话、意图识别、卡片生成 | 旗舰模型 |
| 架构 Agent | 技术方案设计、测试用例 | 旗舰模型 |
| 管理 Agent | 任务拆分、PR 生成 | 中高端模型 |
| 技术栈 Agent | 代码开发、测试运行 | 中端模型（可用本地模型降成本） |
| 测试 Agent | 测试验证、失败修复 | 中端模型 |

#### 2.5.4 模型降级策略

1. 按 Agent 类型匹配模型链配置
2. 若配置的 Provider/模型不可用，自动降级到该 Provider 的默认模型
3. 若 Provider 不可用，尝试下一个可用 Provider
4. 若全部不可用，终止执行并通知用户

#### 2.6.6 领域事件

| 领域 | 事件 | 触发条件 | 消费者 |
|------|------|---------|--------|
| 需求 | `CardCreated` | 新建卡片 | Pipeline, Notify |
| 需求 | `CardConfirmed` | 用户确认卡片 | Pipeline, Notify |
| 需求 | `CardSuperseded` | 卡片被替代 | Pipeline |
| 需求 | `CardAbandoned` | 卡片废弃 | Pipeline |
| 流水线 | `StageStarted` | 阶段开始 | Agent, Notify |
| 流水线 | `StageCompleted` | 阶段完成 | 下一阶段, Notify |
| 流水线 | `StageFailed` | 阶段失败 | Notify |
| 流水线 | `StageReviewRequired` | 需要人工审核 | Notify |
| 智能体 | `TaskAssigned` | 任务分配 | ClonePool, Notify |
| 智能体 | `TaskStarted` | 任务开始执行 | ClonePool |
| 智能体 | `TaskCompleted` | 任务完成 | ClonePool, Pipeline |
| 智能体 | `TaskFailed` | 任务失败 | Notify |
| 智能体 | `TaskInterrupted` | 任务中断（崩溃） | Notify |
| 克隆池 | `CloneAcquired` | 克隆被占用 | Task |
| 克隆池 | `CloneReleased` | 克隆被释放 | — |
| 克隆池 | `CloneBecameDirty` | 克隆产生未提交变更 | Notify |
| AI 运行时 | `ModelCalled` | 模型被调用 | Billing |
| AI 运行时 | `ModelFailed` | 模型调用失败 | Agent, Notify |

**状态流转规则**:

| 转换 | 触发条件 | 说明 |
|------|---------|------|
| 草稿 → 已确认 | 用户主动确认，或沉默 N 分钟自动确认 | 进入需求池 |
| 已确认 → 进行中 | 用户手动启动，或依赖卡完成后自动解锁 | 进入 Pipeline |
| 进行中 → 已完成 | Pipeline 全部阶段完成 | 卡片生命周期结束 |
| 草稿/已确认 → 已变更 | 被新卡片替代 | 旧卡片保留原文，指向新卡片 |
| 草稿/已确认 → 已废弃 | 用户主动废弃 | 不再参与任何流程 |
| 进行中 → 阻塞中 | 依赖卡被变更/废弃 | Pipeline 暂停，等待人工决策 |

### 4.4 批量生成与依赖分析

一次对话可能生成多张卡片，AI 自动分析依赖关系：

**依赖冲突处理**:

### 4.5 变更预览机制

用户描述需求后，AI 生成卡片预览，**只显示净变更**：

**净变更规则**:

| 规则 | 说明 |
|------|------|
| 新增卡片 | 池中不存在 → 预览新增 |
| 变更卡片 | 存在旧卡片且未进入 Pipeline → 预览为"变更"而非"新增+废弃" |
| 废弃卡片 | 存在且未进入 Pipeline → 预览废弃 |
| 过渡卡片自动抵消 | 同一轮对话中引入又废弃的卡片 → 不出现在预览中 |
| 进入 Pipeline 的卡片被覆盖 | 禁止静默覆盖 → 触发特殊警告，需走正式变更 |

### 5.2 各阶段详解

#### 阶段 ①：需求分析

| 维度 | 说明 |
|------|------|
| **输入** | 用户 Vibe 输入 + 知识库上下文 + 当前卡片池 |
| **执行者** | 产品 Agent |
| **输出** | 结构化需求卡片（标题、用户故事、验收标准、优先级） |
| **AI 评审** | 需求完整性检查、与已有卡片的冲突/重叠检测 |
| **人工审核** | **必须** — 用户确认卡片内容（含变更预览） |
| **产出物** | 需求卡片（存入数据库） |

**核心能力**:
- 意图识别：从碎片输入中提取有效需求
- 卡片生成：根据上下文批量生成结构化卡片
- 依赖分析：识别卡片间依赖关系
- 主动澄清：对模糊需求提问，填充验收标准
- 变更检测：新输入覆盖旧卡片时提示用户确认

#### 阶段 ②：方案设计

| 维度 | 说明 |
|------|------|
| **输入** | 需求卡片 + 知识库（架构文档/API 文档） + 开发规范（技术栈） |
| **执行者** | 架构 Agent |
| **输出** | 技术方案（API 设计、数据库设计、架构决策） |
| **AI 评审** | 方案可行性、安全性、与现有架构的兼容性 |
| **人工审核** | 可配置（默认人工确认） |
| **产出物** | `cards/REQ-{ID}/design.md` |

#### 阶段 ③：任务拆解

| 维度 | 说明 |
|------|------|
| **输入** | 技术方案 + 项目仓库结构 + 已有代码上下文 |
| **执行者** | 管理 Agent |
| **输出** | 开发任务列表（按仓库/模块拆分，粒度可被单个 Agent 执行） |
| **AI 评审** | 任务粒度合理性、依赖完整性、工作量估算 |
| **人工审核** | 可配置（默认人工确认，可修改任务拆分） |
| **产出物** | `cards/REQ-{ID}/tasks.md` + 任务记录（数据库） |

**任务拆解原则**:
- 每个任务对应一个明确的代码变更单元
- 同一卡片的同一仓库最多 1 个并行任务（防重）
- 任务粒度：单个 Agent 可在一次执行中完成
- 任务间依赖自动推导

#### 阶段 ④：测试用例

| 维度 | 说明 |
|------|------|
| **输入** | 需求卡片 + 技术方案 |
| **执行者** | 架构 Agent |
| **输出** | 测试用例集（单元测试用例、验收测试用例） |
| **AI 评审** | 覆盖率分析、有效性检查 |
| **人工审核** | **必须** — 质量门禁，通过后才能进入代码开发 |
| **产出物** | `cards/REQ-{ID}/testcases.md` |

#### 阶段 ⑤：代码开发

| 维度 | 说明 |
|------|------|
| **输入** | 开发任务列表 + 测试用例 + 开发规范 + 知识库 + 克隆仓库 |
| **执行者** | 技术栈 Agent（按任务类型路由） |
| **输出** | 源代码 + 单元测试代码 |
| **AI 评审** | 代码质量检查（自动 lint） |
| **人工审核** | 自动（代码开发阶段无需人工介入） |
| **产出物** | 克隆仓库中的代码变更 |

**执行流程**:
1. 管理 Agent 分析任务涉及的仓库
2. 从仓库克隆池获取空闲克隆实例（如无空闲则动态创建）
3. 验证克隆状态为 clean（丢弃所有本地修改，还原到干净状态），创建独立分支
4. 按任务类型路由到对应技术栈 Agent（前端/后端）
5. Agent 在克隆仓库中读写代码，遵循开发规范
6. **无论成功或失败**，必须 commit 并 push 到独立分支
7. 任务完成后归还克隆实例到池中，标记为 idle

**Git 提交策略**：

| 阶段 | 策略 | 说明 |
|------|------|------|
| **开发过程中** | 增量 commit | 每个有意义的变更完成后立即 commit + push，保留细粒度历史 |
| **任务完成时** | 不强制 squash | 中间 commit 保留，便于问题追溯 |
| **PR 合并前** | 可选 squash | 允许用户在合并时选择 squash，保持主干整洁 |

**增量 commit 触发条件**：
- 完成一个文件或一组相关文件的修改
- 测试用例通过
- 完成一个子任务（如实现一个 API、一个组件）

**Commit 消息规范**：
```bash
# 增量 commit 格式（开发中）
feat(req-{id}): {简短描述}
feat(req-{id}): {简短描述}
chore(req-{id}): refactor login handler

# 合并 commit 格式（PR squash 后）
feat(REQ-001): 实现用户登录功能
```

**PR 合并时的 squash 选项**：

```
┌─────────────────────────────────────────────────────────┐
│  创建 Pull Request                                    │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  分支: feature/REQ-001-login                            │
│  → main                                                 │
│                                                          │
│  Commit 历史:                                            │
│  ├─ feat(req-001): init login handler                   │
│  ├─ feat(req-001): add JWT middleware                   │
│  ├─ feat(req-001): implement logout                      │
│  └─ test(req-001): add login tests                      │
│                                                          │
│  Squash Commits:                                         │
│  ○ 不压缩 ─ 保留所有 commit 历史                         │
│  ● 压缩为 1 个 commit ─ 合并为一条简洁记录（推荐）      │
│                                                          │
│  ─────────────────────────────────────────────────────  │
│  [取消]                                     [创建 PR]    │
└─────────────────────────────────────────────────────────┘
```

**Commit 历史示例**：

```
# 不压缩（保留完整历史）
* abc123 feat(req-001): add logout
* def456 feat(req-001): implement logout
* ghi789 feat(req-001): add JWT middleware
* jkl012 feat(req-001): init login handler

# Squash 后（合并为一条）
* mno345 feat(REQ-001): 实现用户登录功能
```

**克隆仓库复用机制**:

```
┌─────────────────────────────────────────────────────────┐
│              仓库克隆池 (按仓库独立管理)                     │
├─────────────────────────────────────────────────────────┤
│  repo: backend                                         │
│  ├─ Clone-001  (idle, clean)   ← 可用                    │
│  ├─ Clone-002  (idle, clean)   ← 可用                    │
│  └─ Clone-003  (busy: TASK-042)  ← 占用中                 │
│                                                         │
│  repo: frontend                                         │
│  ├─ Clone-001  (idle, clean)   ← 可用                    │
│  └─ Clone-002  (busy: TASK-043)  ← 占用中                 │
└─────────────────────────────────────────────────────────┘
```

**克隆实例生命周期**:

| 阶段 | 状态 | 说明 |
|------|------|------|
| 获取 | idle → busy | 从池中获取，验证 clean，创建分支 |
| 执行 | busy | Agent 在克隆中工作 |
| 归还 | busy → idle | commit + push 后，丢弃本地修改，还原 clean |
| 扩容 | — | 池中无空闲实例时，动态克隆新实例 |
| 异常恢复 | — | 系统崩溃后，被占用的克隆标记为 dirty，通过任务卡片状态恢复 |

**任务路由规则**:

| 任务特征 | 路由到 | 说明 |
|---------|--------|------|
| 后端 API / 数据库 | 后端 Agent | Go/Java/Python 等 |
| 前端页面 / 组件 | 前端 Agent | Vue/React 等 |
| 通用脚本 / 配置 | 通用 Agent | 不区分前后端 |
| 跨端变更 | 后端 Agent + 前端 Agent | 并行执行，分别处理各自仓库 |

#### 阶段 ⑥：测试验证

| 维度 | 说明 |
|------|------|
| **输入** | 克隆仓库代码 + 测试用例 |
| **执行者** | 技术 Agent（自动运行） |
| **输出** | 测试报告（通过率、失败详情） |
| **AI 评审** | 分析失败原因，自动修复后重试（最多 3 次） |
| **人工审核** | 自动（失败时通知用户） |
| **产出物** | 测试报告（存入数据库） |

**执行流程**:
1. Agent 在克隆仓库中运行测试命令（根据技术栈配置）
2. 收集测试结果
3. 若有失败，AI 分析失败原因并修复代码（最多 3 轮）
4. 修复后重新运行测试（每次修复后都 commit + push）
5. 3 轮后仍有失败 → 标记任务失败，克隆保留最后状态，通知人工介入

**失败处理原则**:
- 失败也 commit + push，确保克隆状态可追溯
- 系统崩溃时，被占用克隆通过任务状态标记为 interrupted
- 人工可选择：从最后提交继续，或废弃当前任务重新开始

#### 阶段 ⑦：代码评审 & 合并

| 维度 | 说明 |
|------|------|
| **输入** | 克隆仓库的开发分支 |
| **执行者** | 管理 Agent（生成 PR/MR） + 人工（Review + 合并） |
| **输出** | Pull Request / Merge Request，等待人工合并 |
| **AI 评审** | PR 描述生成、代码变更摘要、潜在风险评估 |
| **人工审核** | **必须** — 人工 Review 后手动合并 |
| **产出物** | PR/MR（系统） + 合并记录（人工） |

**PR/MR 创建流程**:

```
克隆仓库 feature/REQ-001-login
           │
           ▼
  ┌─────────────────┐
  │ 自动生成 PR/MR  │
  │ 标题: [REQ-001] │
  │  登录功能开发   │
  │ 关联 REQ-001    │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ Code Review     │ 人工审核
  │ AI 生成变更摘要 │ + CI 检查
  └────────┬────────┘
           │ 人工合并
           ▼
  ┌─────────────────┐
  │ 人工合并到      │
  │ main/master    │ ← 系统不执行合并
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ 归还克隆到池    │ 丢弃分支，还原 clean
  │ 卡片标记完成    │
  └─────────────────┘
```

**系统边界**：
- ✅ 系统负责创建 PR/MR
- ✅ 系统负责生成 PR 描述和变更摘要
- ❌ 系统不执行合并操作
- ❌ 系统不删除开发分支

### 5.3 阶段审核配置

每个阶段独立配置审核模式。**AI 评审不可跳过**，差异仅在最终放行权归谁：

| 审核模式 | AI 评审 | 最终确认 | 说明 |
|----------|---------|---------|------|
| **ai_auto** | ✅ 必须 | AI 通过即放行 | 最快，无需人工 |
| **human** | ✅ 必须 | 人工确认后才放行 | 最严，质量优先 |
| **skip** | ❌ 跳过 | ❌ 跳过 | 跳过该阶段，谨慎使用 |

**各阶段默认审核模式**:

| 阶段 | 默认模式 | 说明 |
|------|---------|------|
| ① 需求分析 | human | 卡片确认必须人工 |
| ② 方案设计 | human | 技术方案需人工确认 |
| ③ 任务拆解 | human | 任务拆分可修改 |
| ④ 测试用例 | human | 质量门禁，不可跳过 |
| ⑤ 代码开发 | ai_auto | Agent 自动执行 |
| ⑥ 测试验证 | ai_auto | 自动运行，失败时通知 |
| ⑦ 评审合并 | human | 合并必须人工确认 |

### 5.4 Pipeline 配置

每个项目可在 `pipeline.yaml` 中自定义：

    - id: "design"

      review_mode: "human"

    - id: "task_breakdown"
      name: "任务拆解"
      enabled: true
      review_mode: "human"
      ai_review: true

    - id: "testcase"
      name: "测试用例"
      enabled: true
      review_mode: "human"          # 质量门禁
      ai_review: true

    - id: "development"

    - id: "testing"

### 5.5 卡片启动与排队

用户确认卡片后，可随时启动进入 Pipeline：

```
用户: 启动 REQ-042
         ↓ 依赖检查
REQ-042 依赖 REQ-037（进行中）
         ↓
进入执行队列（排队中）
         ↓ REQ-037 完成后
自动解锁 → 按序执行

用户: 批量启动 REQ-042、REQ-043、REQ-044
         ↓ 依赖拓扑分析
批次 1: REQ-042（无依赖）→ 执行
批次 2: REQ-043（依赖 042）→ 等待批次 1
批次 3: REQ-044（依赖 043）→ 等待批次 2
```

**执行规则**:

| 规则 | 说明 |
|------|------|
| 随时启动 | 已确认卡片可随时进入排队 |
| 拓扑排序 | 按依赖关系依次执行（Kahn 算法） |
| 1:1 任务约束 | 同一卡片的同一仓库只能有 1 个执行中任务 |
| 可取消 | 未开始执行的卡片可随时取消 |
| 不可中断 | 已开始 Pipeline 的卡片不可中断，必须完成或降级 |

## 6. Agent 矩阵

### 6.1 Agent 类型

NBCoder 定义了四类逻辑 Agent。**子 Agent 封装为 Skill**，以 tool 方式调用。

#### 产品 Agent：受限的 OpenClaw

产品 Agent 是一个**受限版本的 OpenClaw**，专注于产品需求工作流：

| 特性 | 说明 |
|------|------|
| **工具范围** | 仅限产品文档读写、卡片管理 |
| **能力边界** | 不执行代码、不操作 Git、不调度其他 Agent |
| **核心输出** | 需求卡片（待决策状态） |
| **对话目标** | 理解用户意图，生成结构化需求 |

**产品 Agent 工具白名单**：

| 工具 | 权限 |
|------|------|
| `read_file` | ✅ 仅限产品文档目录 |
| `write_file` | ✅ 仅限产品文档目录 |
| `search_file` | ✅ 仅限产品文档目录 |
| `search_content` | ✅ 仅限产品文档目录 |
| `create_card` | ✅ 创建需求卡片 |
| `update_card` | ✅ 更新卡片草稿 |
| `query_cards` | ✅ 查询卡片池 |
| `git_*` | ❌ 禁止 |
| `run_*` | ❌ 禁止 |
| `pr_*` | ❌ 禁止 |

**产品 Agent 的边界**：
```
用户对话
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│                 产品 Agent（受限 OpenClaw）               │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ✅ 可做                                                │
│  ├─ 理解用户需求（自然语言）                              │
│  ├─ 生成/编辑 PRD 文档                                   │
│  ├─ 生成需求卡片（草稿状态）                              │
│  ├─ 分析卡片依赖关系                                     │
│  └─ 访问知识库文档                                       │
│                                                          │
│  ❌ 不可做                                               │
│  ├─ 直接生成代码                                          │
│  ├─ 执行命令/脚本                                         │
│  ├─ 操作 Git/仓库                                        │
│  ├─ 触发 Pipeline 执行                                   │
│  └─ 调度其他 Agent                                       │
│                                                          │
└─────────────────────────────────────────────────────────┘
    │
    │ 用户审核通过后
    ▼
系统自动触发后续流程（架构 Agent、管理 Agent 等）
```

**逻辑 Agent（用户可见）**：

| Agent | 定位 | 交互方式 | Pipeline 阶段 |
|-------|------|---------|-------------|
| **产品 Agent** | 与用户直接对话 | 用户直接对话触发 | 需求分析、卡片生成、依赖分析 | ① 需求分析 |
| **架构 Agent** | 系统调度后台执行 | 系统自动触发 | 技术方案设计、测试用例编写 | ② 方案设计、④ 测试用例 |
| **管理 Agent** | 系统调度后台执行 | 系统自动触发 | 任务拆解、克隆池管理、PR 生成 | ③ 任务拆解、⑦ 评审合并 |
| **技术栈 Agent** | 系统调度后台执行 | Tool 调用 Skill | 并行代码开发、测试运行 | ⑤ 代码开发、⑥ 测试验证 |

**实现细节（隐藏子 Agent）**：

| 逻辑 Agent | 可能的隐藏子 Agent |
|-----------|------------------|
| 产品 Agent | 对话 Agent、意图识别 Agent、卡片生成 Agent、依赖分析 Agent |
| 架构 Agent | API 设计 Agent、数据库设计 Agent、架构决策 Agent |
| 管理 Agent | 任务拆分 Agent、依赖分析 Agent、PR 生成 Agent |
| 技术栈 Agent | 前端 Agent、后端 Agent、测试 Agent、lint Agent |

**说明**：
- 隐藏子 Agent 由系统自动创建，用户无感知
- 子 Agent 之间相互隔离，独立上下文
- 对外只暴露逻辑 Agent 的执行结果

### 6.2 Agent 架构

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   用户 ──── 对话 ────→ 产品 Agent                           │
│                              │                              │
│                              │ 用户审核通过                   │
│                              ↓                              │
│   ┌─────────────────────────────────────────────────────┐   │
│   │                    系统自动调度                        │   │
│   │                                                      │   │
│   │   管理 Agent ──→ spawn ──→ 技术栈 Agent             │   │
│   │                                                      │   │
│   │   架构 Agent（方案设计 / 测试用例）                    │   │
│   │                                                      │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**核心设计原则**:

| 原则 | 说明 |
|------|------|
| **子 Agent = Skill** | 所有子 Agent 封装为 Skill，以 tool 方式调用 |
| **入口 Agent = 轮询触发** | 入口 Agent 由任务调度器轮询任务表触发 |
| **无实时性要求** | 采用批量处理模式，轮询间隔可配置 |
| **Skill 可复用** | 同一 Skill 可被多个入口 Agent 调用 |

**调度关系**:
- 产品 Agent：**用户对话触发**，不调度其他 Agent，仅生成需求卡片
- 入口 Agent：**任务调度器轮询触发**，执行 Pipeline 各阶段
- Skill：**以 tool 方式调用**，封装具体能力

**子 Agent（系统调度 Agent）特性**:

| 特性 | 说明 |
|------|------|
| 任务轮询触发 | 入口 Agent 由任务调度器轮询任务表触发 |
| Skill 调用 | 通过 tool 方式调用封装好的 Skill |
| 独立上下文 | 按需注入，不继承其他 Agent 记忆 |
| 独立模型配置 | 可配置不同模型（Claude 写代码、GPT-4o 写文档） |
| 工具白名单 | 每类 Agent 可访问的工具范围不同 |
| 执行完毕后回收 | 完成任务后销毁，不长期驻留 |
| 超时控制 | 单次执行有最大时长限制 |

#### 6.2.1 任务调度器

任务调度器负责轮询任务表，触发入口 Agent 执行：

```

### 6.6 产品 Agent 对话记忆

#### 6.6.1 三层记忆模型

产品 Agent 采用 OpenClaw 的三层记忆模型管理对话历史，确保关键信息跨会话持久化：

```
┌─────────────────────────────────────────────────────────────────┐
│                    三层记忆模型                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Layer 1: Context（工作记忆）                             │   │
│  │  - 当前会话的所有消息                                     │   │
│  │  - 受 context window 限制                                │   │
│  │  - 对话越长，早期信息越容易被挤出                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                   │
│                              │ Compaction 触发                   │
│                              ▼                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Layer 2: Compaction（压缩摘要）                          │   │
│  │  - 当 context 接近上限时自动压缩                          │   │
│  │  - 旧对话总结为摘要，保留要点丢弃细节                      │   │
│  │  - 摘要持久化到会话历史                                   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                   │
│                              │ Memory Flush                       │
│                              ▼                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Layer 3: Memory Files（持久化记忆）                      │   │
│  │  - MEMORY.md：策展的长期记忆（偏好、决策、教训）          │   │
│  │  - memory/YYYY-MM-DD.md：每日操作日志                     │   │
│  │  - 唯一跨会话存在的记忆层                                  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### 6.6.2 Compaction 压缩机制

**触发条件**：
```
当 token 数 > (context_window - 20000 - 4000) 时触发 compaction
```

**压缩流程**：

```
1. Memory Flush（压缩前）
   └─→ 提醒 Agent 将重要信息写入 memory 文件
   └─→ 关键决策 → MEMORY.md
   └─→ 日常记录 → memory/YYYY-MM-DD.md

2. Compaction 执行
   └─→ 将早期对话总结为结构化摘要
   └─→ 保留：决策、结论、待办
   └─→ 丢弃：闲聊、已解决的探索过程

3. 摘要持久化
   └─→ 写入会话历史文件
   └─→ 未来请求用摘要替代原始消息
```

**压缩前静默轮次**：

```yaml
# 压缩前静默轮次配置
compaction:
  reserveTokensFloor: 20000      # 保留 token 下限
  memoryFlush:
    enabled: true                 # 启用内存刷新
    softThresholdTokens: 4000    # 提前提醒阈值
    triggerPrompt: |
      会话即将压缩。请将重要信息保存到 memory 文件：
      - 关键决策 → MEMORY.md
      - 今日进展 → memory/YYYY-MM-DD.md
      如无重要信息，回复 NO_REPLY。
```

### 8.1 概述

开发规范是约束 AI Agent 开发行为的配置文件，确保 Agent 产出的代码符合项目约定。

### 8.2 规范内容

| 规范项 | 存储位置 | 格式 | 说明 |
|--------|---------|------|------|
| 技术栈定义 | `.NBCoder/standards/tech-stack.md` | Markdown | 前后端技术栈、框架版本、依赖约束 |
| 编码规范 | `.NBCoder/standards/coding-rules.md` | Markdown | 命名规则、代码风格、目录结构约定 |
| Commit 格式 | `.NBCoder/standards/commit-convention.md` | Markdown | Commit message 格式要求 |
| 分支策略 | `pipeline.yaml` | YAML | 分支命名模板（已在 Pipeline 配置中） |

### 8.3 规范示例

```markdown
<!-- .NBCoder/standards/tech-stack.md -->
# 技术栈

## 后端
- 语言: Go 1.22+
- 框架: Gin
- 数据库: SQLite
- ORM: GORM

## 前端
- 框架: Vue 3.4+
- 语言: TypeScript 5.x
- UI 库: Element Plus
- 构建工具: Vite
```

```markdown
<!-- .NBCoder/standards/coding-rules.md -->
# 编码规范

## 命名规则
- 函数/方法: 驼峰式 (camelCase)
- 结构体/接口: 大驼峰 (PascalCase)
- 常量: 全大写下划线 (SCREAMING_SNAKE_CASE)
- 文件名: 小写下划线 (snake_case)

## 错误处理
- 所有 error 必须显式处理，禁止 `_ = someFunc()`
- 错误信息使用英文，首字母大写
- 自定义错误需包装上下文: fmt.Errorf("do something: %w", err)

## 目录结构
- 每个功能模块一个独立目录
- handler / service / repository 三层分离

## Git 提交规范

### 开发中 Commit（增量）
每次完成有意义的变更后立即提交：

```bash
# 格式
<type>(req-{id}): <简短描述>

# 示例
feat(req-001): init login handler
feat(req-001): add JWT middleware
test(req-001): add login tests
chore(req-001): refactor code style
fix(req-001): handle empty password
```

### Type 类型
| Type | 说明 | 使用场景 |
|------|------|---------|
| feat | 新功能 | 新增功能代码 |
| fix | 修复 | Bug 修复 |
| refactor | 重构 | 代码重构不改变功能 |
| test | 测试 | 新增/修改测试用例 |
| docs | 文档 | 文档更新 |
| chore | 杂项 | 构建、依赖更新等 |
| perf | 性能 | 性能优化 |

### PR Merge Commit（最终合并）
通过 GitHub/GitLab squash 合并时，生成最终 commit：

```bash
# 格式
<type>(REQ-{id}): {功能描述}

# 示例
feat(REQ-001): 实现用户登录功能
```

**注意**：
- 开发中 commit 保留完整历史，便于追溯每个变更
- PR 合并时由用户选择是否 squash，保持主干整洁

| 文件操作 | < 100ms |

### 12.2 可靠性

- 服务崩溃自动恢复（启动时检测 interrupted 任务，修复克隆状态）
- 卡片自动保存
- Agent 执行记录完整持久化
- 变更历史可查
- 项目数据备份机制（数据库备份 + 文件系统备份）

**崩溃检测与告警**（可选增强）：

| 机制 | 说明 |
|------|------|

| 外部进程管理 | K8s livenessProbe / systemd watchdog / supervisor |

### 12.4 服务生命周期管理

#### 12.4.0 崩溃恢复机制

意外崩溃（如断电、OOM kill、进程 crash）无法优雅关闭，启动时必须进行状态恢复：

```
┌─────────────────────────────────────────────────────────────┐
│                    启动时崩溃恢复流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────────────┐                                       │
│   │ 1. 锁定数据库    │  防止并发修复                          │
│   └────────┬────────┘                                       │
│            │                                                │
│            ▼                                                │
│   ┌─────────────────┐                                       │
│   │ 2. 检测异常状态   │  查询 running/interrupted 任务         │
│   └────────┬────────┘                                       │
│            │                                                │
│            ▼                                                │
│   ┌─────────────────┐                                       │
│   │ 3. 恢复克隆实例   │  遍历 busy 克隆，验证/修复状态         │
│   └────────┬────────┘                                       │
│            │                                                │
│            ▼                                                │
│   ┌─────────────────┐                                       │
│   │ 4. 通知用户      │  告知崩溃影响范围和待处理事项            │
│   └────────┬────────┘                                       │
│            │                                                │
│            ▼                                                │
│   ┌─────────────────┐                                       │
│   │ 5. 解锁数据库    │  恢复服务正常运行                       │
│   └─────────────────┘                                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**任务状态修复策略**：

| 原状态 | 修复方式 | 说明 |
|--------|---------|------|
| `running` | → `interrupted` | 任务被强制中断，克隆状态未知 |
| `queued` | 保持 `queued` | 未开始执行，不受影响 |
| `pending` | 保持 `pending` | 未开始执行，不受影响 |

**克隆实例修复策略**：

| 克隆状态 | 修复方式 | 说明 |
|---------|---------|------|
| `busy` + 关联任务 interrupted | → `idle` + `dirty` | 克隆可能有未提交变更，需人工检查 |
| `busy` + 无关联任务 | → `idle` | 孤儿克隆，直接归还 |
| 克隆目录不存在 | 删除记录 | 物理克隆已丢失 |

**克隆 dirty 恢复流程**：

```
克隆标记为 dirty
       │
       ▼
┌─────────────────────────┐
│ 1. 尝试 git status      │  检查是否有未提交变更
└────────┬────────────────┘
         │
    ┌────┴────┐
    │ 有变更？ │
    └────┬────┘
    是   │   否
    ▼    │    ▼
┌────────┐ │ ┌────────────────┐
│ 通知   │ │ │ git checkout . │
│ 用户   │ │ │ git clean -fd  │
│ 人工   │ │ │ 还原 clean     │
│ 处理   │ │ └────────────────┘
└────────┘ │
           ▼
    ┌────────────────┐
    │ 标记为 idle    │
    └────────────────┘
```

#### 12.4.1 优雅关闭

服务关闭或重启时，必须保证运行中的任务安全完成，防止数据损坏和资源泄漏：

```
┌─────────────────────────────────────────────────────────────┐
│                  服务关闭流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────────────┐                                       │
│   │ 1. 接收关闭信号   │  SIGTERM / SIGINT / 系统重启触发      │
│   └────────┬────────┘                                       │
│            │                                                │
│            ▼                                                │
│   ┌─────────────────┐                                       │
│   │ 2. 停止接收新任务 │  任务调度器暂停，不分发新任务           │
│   └────────┬────────┘                                       │
│            │                                                │
│            ▼                                                │
│   ┌─────────────────┐                                       │
│   │ 3. 安全检查       │  验证所有运行中任务状态               │
│   └────────┬────────┘                                       │
│            │                                                │
│            ▼                                                │
│   ┌─────────────────┐                                       │
│   │ 4. 等待任务完成   │  等待 running 任务正常完成            │
│   │    或超时处理    │  超时后强制终止并记录状态              │
│   └────────┬────────┘                                       │
│            │                                                │
│            ▼                                                │
│   ┌─────────────────┐                                       │
│   │ 5. 清理资源      │  关闭数据库连接、释放克隆实例           │
│   └────────┬────────┘                                       │
│            │                                                │
│            ▼                                                │
│   ┌─────────────────┐                                       │
│   │ 6. 退出服务       │  正常退出（exit code = 0）            │
│   └─────────────────┘                                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### 12.4.2 安全检查机制

**任务状态检查**：

| 任务状态 | 关闭时的行为 |
|---------|------------|
| `pending` | 取消执行，保持 pending 状态 |
| `queued` | 取消执行，保持 queued 状态 |
| `running` | **等待完成**，设置最大等待时间 |
| `completed` | 已完成，无需处理 |
| `failed` | 已失败，无需处理 |
| `interrupted` | 已中断，无需处理 |

**等待策略**：

```yaml
graceful_shutdown:
  wait_running_tasks: true           # 等待运行中任务完成
  max_wait_duration: 30s             # 最大等待时间
  force_kill_after_timeout: true     # 超时后强制终止
  force_kill_timeout: 10s            # 强制终止前的宽限期
```

**异常恢复**：

| 场景 | 处理方式 |
|------|---------|
| 服务崩溃 | 启动时检测 `running` 状态的任务，标记为 `interrupted` |
| 任务超时 | 强制终止，克隆状态标记为 dirty，通知用户人工介入 |
| 克隆实例占用 | 重启后归还到 clone_pool，验证 clean 后放回 idle |

#### 12.4.3 关闭事件触发源

| 触发源 | 说明 |
|--------|------|
| `SIGTERM` | Kubernetes / Docker 停止信号 |
| `SIGINT` | Ctrl+C 本地终止 |
| `systemctl stop` | systemd 服务停止 |
| API 调用 | `/api/shutdown` 端点（可选） |
| WebSocket 指令 | 管理界面发起关闭 |

#### 12.4.4 克隆池关闭保护

```
关闭信号
     │
     ▼
┌─────────────────────────────────────────────┐
│  遍历所有 clone_instances (status = busy)    │
├─────────────────────────────────────────────┤
│                                             │
│  ┌───────────────────────────────────────┐  │
│  │ 对每个 busy 实例：                     │  │
│  │ 1. 检查关联任务状态                      │  │
│  │ 2. 如果 running → 等待完成              │  │
│  │ 3. 执行 git stash / git checkout clean  │  │
│  │ 4. 提交最后状态到日志                    │  │
│  │ 5. 标记为 idle                          │  │
│  └───────────────────────────────────────┘  │
│                                             │
│  所有实例 clean 后 → 服务退出                │
└─────────────────────────────────────────────┘
```

## 13. 技术实现说明

> 以下内容为开发阶段的技术设计指导，具体实现时可根据技术选型调整。

### 13.1 存储方案

每个项目拥有独立的数据库文件，位于项目元数据目录下，项目间完全隔离。

**存储选型原则**：
- 项目数据完全隔离，单个项目的数据库损坏不影响其他项目
- 备份简单，拷贝项目目录即可完整备份
- 私有化部署，零运维成本

### 13.2 核心数据

| 数据类型 | 存储位置 | 说明 |
|---------|---------|------|
| 项目元数据 | 项目元数据目录 | 配置文件 |
| 需求卡片 | 数据库 | 卡片、依赖、状态 |
| 开发任务 | 数据库 | 任务、状态、执行记录 |
| 克隆池 | 数据库 + 文件系统 | 实例信息、本地仓库 |
| 知识库 | 文件系统 | Markdown 文档 |
| AI 配置 | YAML 配置文件 | Provider、模型链 |
| Agent 日志 | 文件系统 | JSON Lines 日志文件 |

### 13.3 领域边界（DDD）

系统采用领域驱动设计，划分为以下限界上下文：

| 上下文 | 核心实体 | 说明 |
|--------|---------|------|
| 需求上下文 | Card, CardDependency | 卡片生命周期、依赖关系 |
| 流水线上下文 | Pipeline, Stage | 阶段编排、审核控制 |
| 智能体上下文 | Task, Skill | 任务调度、执行记录 |
| 克隆池上下文 | CloneInstance | 克隆资源管理 |
| AI 运行时上下文 | Provider, Model | AI 配置与调用 |
| 项目上下文 | Project | 多项目隔离 |

|--------|------|

## 14. 附录

### 14.1 术语表

| 术语 | 说明 |
|------|------|
| 需求卡片 | 独立的需求单元，包含原始表达、结构化产物、依赖关系和生命周期状态 |
| 卡片池 | 单个项目内所有需求卡片的集合 |
| 项目 | 独立的数据隔离单元，拥有独立的知识库、规范、卡片池 |
| 工作根目录 | 通过 `--workdir` 指定的根目录，包含多个项目 |
| Pipeline | 7 阶段开发流水线，卡片从需求分析到评审合并的完整流程 |
| Agent | AI 代理，负责特定阶段的任务执行 |
| 知识库 | 项目级的文档集合，为 AI Agent 提供项目上下文 |
| 开发规范 | 约束 AI Agent 开发行为的配置（技术栈、编码规范等） |
| 血缘 | 产出物间的上下游关系，支持全链路追溯 |
| Vibe 输入 | 自由、口语化、碎片化的需求输入方式 |
| 任务 | 需求拆解后的最小可执行单元，由 Agent 自动执行 |
| 工具白名单 | 每类 Agent 可使用的工具范围 |
| AI Provider | AI 模型服务提供商，如 OpenAI、Claude、本地模型 |
| Agent 模型链 | 为不同类型 Agent 配置的模型分配规则，决定每个 Agent 使用哪个模型 |
| 三层记忆模型 | OpenClaw 的记忆管理机制：Context（工作记忆）→ Compaction（压缩摘要）→ Memory Files（持久化记忆） |
| Compaction | 当 context 接近上限时自动压缩旧对话为摘要的过程 |
| Memory Flush | Compaction 前将重要信息写入 memory 文件的静默轮次 |
| MEMORY.md | 策展的长期记忆文件，存储用户偏好、关键决策、经验教训 |
| memory/YYYY-MM-DD.md | 每日操作日志，记录当日对话摘要、进展、待办 |
| 受限 OpenClaw | 产品 Agent 的定位，仅能修改产品文档和生成需求卡片，不具备代码执行、Git 操作、Agent 调度等能力 |
| Skill | 子 Agent 的封装形式，包含执行指令、工具权限、模型推荐，以 tool 方式被入口 Agent 调用 |
| 任务调度器 | 轮询任务表触发入口 Agent 执行的组件，无实时性要求，采用批量处理模式 |
| 轮询间隔 | 任务调度器扫描任务表的时间间隔，可配置（建议 5s-30s） |
| 优雅关闭 | 服务关闭时保证运行中任务安全完成的机制，先暂停新任务，等待运行中任务完成后清理资源 |
| 安全检查 | 关闭前验证所有任务状态，确保 running 任务已完成或超时处理 |
| 崩溃恢复 | 服务意外终止后，启动时自动修复异常状态的机制，包括 interrupted 任务和 dirty 克隆 |
| dirty 克隆 | 克隆实例存在未提交变更，需要人工处理或还原 |
| 增量 commit | 每次完成有意义的变更后立即提交，保留细粒度历史 |
| Squash | 将多个 commit 合并为一个，常用于 PR 合并时保持主干整洁 |
| DDD | 领域驱动设计（Domain-Driven Design），通过划分限界上下文和聚合来管理复杂业务 |
| 核心域 | 系统中唯一重要的业务领域，是产品差异化的核心，NBCoder 中为需求、流水线、智能体三个领域 |
| 支撑域 | 为核心域提供支撑能力，但不直接创造价值，如知识库、克隆池、AI 运行时 |
| 通用域 | 可跨项目复用的通用能力，如项目、Git、通知 |
| 限界上下文 | Bounded Context，DDD 中划分业务边界的核心概念，每个上下文有独立的领域模型 |
| 聚合 | Aggregate，一组相关对象的组合，作为数据修改的单元，有明确的边界和根实体 |
| 防腐层 | Anti-Corruption Layer，在上下文映射中，保护本上下文不被外部上下文的变化影响 |
| 领域事件 | Domain Event，领域中发生的业务事件，用于解耦和追踪业务流程 |
